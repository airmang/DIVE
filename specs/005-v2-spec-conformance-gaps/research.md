# Research: V2 Spec Conformance Gaps

## Decision 1: Unsupported Runtime Means Capability Block, Not Legacy Fallback

**Decision**: In v2 user work, providers without confirmed Pi capability and
local legacy runtime requests are represented as explicit unavailable/capability
states. They do not execute the legacy loop.

**Rationale**: The constitution and foundation spec define Pi as the only v2
runtime. Keeping fallback selection would make runtime state, permission
supervision, and classroom behavior unauditable.

**Alternatives considered**:

- Keep legacy as a hidden fallback for unsupported providers. Rejected because
  it directly conflicts with the v2 runtime boundary.
- Keep a developer-only override. Rejected for user-visible work because saved
  preferences or environment state can leak into classroom behavior.
- Remove old AgentLoop code entirely in this slice. Deferred because the safer
  first step is preventing user-visible selection while preserving unrelated
  migration/test surfaces until a later cleanup.

## Decision 2: Scope Expansion Is A New Supervisor Event

**Decision**: Add-step scope expansion is a deterministic DIVE event that can
invoke the SupervisorAgent after DIVE builds evidence refs from PRD criteria,
PRD scope, add-step fields, and reason codes.

**Rationale**: DIVE should own the trigger and evidence, while the
student-facing question belongs to the dedicated SupervisorAgent. This matches
the review-card architecture and avoids frontend-authored static questions.

**Alternatives considered**:

- Keep the local hardcoded scope card. Rejected because active guidance says
  review-card questions are generated by SupervisorAgent, not frontend rules.
- Treat scope expansion as `verify_entered`. Rejected because the artifact and
  evidence are an add-step draft, not final verification.
- Wait for a broad future event system. Rejected because this is a small,
  concrete conformance gap with existing scope evidence.

## Decision 3: Supervisor Failure Produces No Scope Card

**Decision**: Scope-expansion supervisor timeout, unavailable model, malformed
decision, invalid evidence, duplicate decision, or disallowed actions produces
no card plus EventLog/export record.

**Rationale**: Static fallback cards are explicitly out of scope. Silence with
audit logging is safer than showing generic or unvalidated warnings.

**Alternatives considered**:

- Reuse the existing frontend card as fallback. Rejected as static fallback.
- Block add-step save while waiting for supervisor. Rejected because the card is
  a low-friction review annotation, not a mutation gate.

## Decision 4: Rationale Challenge Offer Is Deterministic And Non-Blocking

**Decision**: A valid step-rationale objection records the objection and creates
an `offered` plan-adjustment next action. The offer may seed a plan-area change
proposal, but no mutation is applied until the student confirms it in the
dedicated plan area.

**Rationale**: This satisfies the success criterion that challenge offers
re-decomposition without turning the objection into an automatic plan rewrite.
It also keeps Work Mode low-friction.

**Alternatives considered**:

- Call an LLM immediately to re-decompose. Deferred because the first required
  behavior is an offer, not automatic generation.
- Log objection only. Rejected because the active success criterion requires an
  offer.
- Mutate the plan immediately after objection. Rejected because plan mutations
  must happen through the dedicated plan area.

## Decision 5: Change-Step And Retire-Step Stay Future/Reserved

**Decision**: This cleanup does not implement visible `change_step` or
`retire_step` mutation flows. Active docs should label them as future or
contract-reserved unless a later feature defines those flows.

**Rationale**: The current add-step requirement is independently valuable and
already has a dedicated surface. Adding change/retire UI would expand product
scope beyond the conformance gaps the user asked to make actionable.

**Alternatives considered**:

- Implement all mutation types now. Rejected as a broader planning/redesign
  slice.
- Leave active docs ambiguous. Rejected because future agents may treat
  reserved contracts as shipped behavior.

## Decision 6: Spec Status Is Part Of Completion

**Decision**: Completion includes updating canonical spec index/status docs and
affected feature notes so implementation truth is visible before future work.

**Rationale**: The discovered gaps came partly from checked-off task lists that
overstated conformance. Status documents need to carry the final state.

**Alternatives considered**:

- Update code only. Rejected because DIVE uses Spec Kit docs as source of truth.
- Move all old specs into this feature. Rejected because 001-004 remain active
  authorities; 005 should clarify and close gaps, not replace them.
