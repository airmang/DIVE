# 작업 33: 접근성 폴리싱 + 색상 대비 검증

## 컨텍스트
작업 32에서 구조적 접근성을 마쳤다면, 이 작업은 시각적 접근성. 색상 대비, 색맹 친화 디자인, 텍스트 크기, 모션 감소.

## 이번 작업 범위
- WCAG AA 준수 — 본문 4.5:1, 액션 버튼 3:1
- 색상만으로 의미 전달하지 않음 — 색 + 아이콘 + 텍스트 동시
- 색맹 시뮬레이션 — 적록·청황 모두 식별 가능
- 모션 감소 옵션 (`prefers-reduced-motion`)
- 텍스트 zoom 200% 동작

## 명세 참조
- DIVE_SPEC.md §12.2 — 색상 대비 (WCAG AA)
- DIVE_SPEC.md §2.3 — 컬러 팔레트

## 단계

1. **색상 대비 검증**:
   - axe DevTools 또는 `eslint-plugin-jsx-a11y` 활용
   - 다크 모드:
     - 본문 (#E8E6F0 on #1A1A1F) — 14.7:1 OK
     - 보조 텍스트 (#8E8C9A on #1A1A1F) — 4.92:1 OK
     - 액센트 버튼 (#B19CD9 on bg) — 검증 필요
   - 라이트 모드 동등 검증
   - 미달 색상 보정 (디자인 토큰 갱신)
2. **색상 외 의미 전달**:
   - 권한 카드 — 색 + 아이콘 + "안전/주의/위험" 텍스트
   - 카드 상태 — 색 + 아이콘 (○ ◐ ◑ ✓) + 텍스트
   - DIVE 진행 점 — 색 + "DIVE" 글자 라벨
   - 진행률 바 — 색 + 숫자 라벨 ("40%")
3. **색맹 시뮬레이션**:
   - Chrome DevTools 색맹 시뮬레이션 (Protanopia, Deuteranopia, Tritanopia, Achromatopsia)
   - 각 모드에서 시나리오 A 따라가기 — 식별 가능 여부
   - 미달 시 컬러 토큰 보정 또는 추가 패턴 (선·점선·아이콘 등)
4. **모션 감소**:
   - `prefers-reduced-motion: reduce` 미디어 쿼리 감지
   - 슬라이드 인 트랜지션 제거 또는 즉시 표시
   - 카드 호버 효과 제거
   - 스피너 → 정적 진행 표시
5. **텍스트 zoom 200%**:
   - 브라우저 zoom 200% 시 레이아웃 깨지지 않음
   - 가로 스크롤 발생 시 OK (워크맵), 세로 스크롤 발생 OK (채팅)
   - 글자 잘림 X (텍스트 컨테이너 max-width 적용)
6. **포커스 시각**:
   - 모든 인터랙티브 요소의 focus ring 명확
   - 색상 (액센트) + 굵기 (2px outline) + 오프셋
7. **고대비 모드** (Windows 고대비):
   - 시스템 고대비 활성화 시 색상 자동 적용 (`forced-colors`)
   - 핵심 정보 표시 유지
8. 단위 테스트 — axe-core integration

## 완료 조건
- [ ] 모든 텍스트 WCAG AA 통과
- [ ] 색상 외 의미 전달이 충분 (색맹 시뮬레이션 통과)
- [ ] `prefers-reduced-motion` 시 애니메이션 제거
- [ ] zoom 200% 시 레이아웃 정상
- [ ] focus ring 모든 인터랙티브 요소에 명확
- [ ] Windows 고대비 모드에서 사용 가능
- [ ] axe-core 자동 검사 0 위반

## 확인 질문
- WCAG AAA 추가 목표 — AA보다 엄격 (7:1). 학습 환경 적합성. AA로 충분 (명세 명시).
- 다크 모드 본문 색상 — `#E8E6F0` 약간 차갑게 보임. 더 따뜻한 톤 검토? 검토만, 명세 변경 시 ADR.
- 포커스 색상 — 액센트(파스텔 보라)가 다크 배경에서 충분히 두드러지는지? 검증 필요.
- 고대비 모드 — `forced-colors: active` CSS는 색상을 OS 색으로 강제. DIVE 디자인 손상되지만 핵심 정보 표시는 유지되어야.
