# 작업 23: 파일럿 환경 검증 (학교 PC)

## 컨텍스트
실제 광교고 PC와 학생 노트북에서 빌드·실행이 정상 동작하는지 사전 검증. 파일럿 직전에 발견하는 문제는 데드라인 위험. 이 작업은 코드 작업이 아니라 검증·체크리스트 중심.

## 이번 작업 범위
- 학교 PC 사양 점검 (CPU·RAM·OS·.NET 등)
- 빌드 결과물(NSIS) 학교 PC 설치 테스트
- OpenRouter 자식 키 25명 동시 시뮬레이션
- 차시 종료 후 export 동작
- 발견된 문제별 hotfix
- 운영 체크리스트 작성

## 명세 참조
- DIVE_SPEC.md §13.5 — 8월 초 광교고 파일럿
- DIVE_SPEC.md §12.1 — 성능
- DIVE_SPEC.md §11.3 — Windows x64 + ARM64 빌드

## 단계

1. **사양 점검** — 광교고 IT 담당자에게 사양 정보 요청:
   - CPU (x64/ARM64)
   - RAM 최소·평균
   - OS 버전 (Windows 10·11)
   - .NET / WebView2 설치 여부 (Tauri는 WebView2 필요)
   - 네트워크 환경 (방화벽, 프록시, 외부 API 차단)
2. **NSIS 인스톨러 검증**:
   - 학교 PC 1대에 직접 설치 (관리자 권한 없는 환경 가정)
   - 시작 시간 측정 (명세 §12.1 — 차가운 시작 3초 이내)
   - 메모리 사용량 측정 (유휴 250MB 이하)
   - WebView2 미설치 환경에서 자동 설치 동작 확인
3. **OpenRouter 25명 시뮬레이션** — 자식 키 25개 발급 → 동시 호출 부하:
   - 시뮬레이션 스크립트 (`scripts/simulate_classroom.ts`)
   - 25개 키로 동시 LLM 호출 50회씩
   - 한도 도달 시 친절한 에러 표시 확인
   - 차시 종료 후 일괄 폐기 확인
4. **차시 종료 후 export 동작** — 위 시뮬레이션 데이터로 export 생성, 분석 가능 형식인지 확인
5. **발견 문제 hotfix** — 빌드·실행·네트워크·성능 이슈 발견 시 즉시 수정. 큰 변경은 별도 ADR.
6. **운영 체크리스트 작성** (`docs/pilot_checklist.md`):
   - 차시 시작 전 — 메인 키 등록, 자식 키 발급, QR 인쇄
   - 차시 중 — 한도 모니터링
   - 차시 종료 후 — 자식 키 폐기, export 생성

## 완료 조건
- [ ] 학교 PC 사양 정보 수집 완료
- [ ] NSIS 인스톨러가 학교 PC에서 정상 설치·실행
- [ ] 시작 시간 3초 이내, 메모리 250MB 이하
- [ ] 25명 동시 시뮬레이션 통과 (큰 에러 없음)
- [ ] 차시 종료 후 export 정상
- [ ] 운영 체크리스트 문서화
- [ ] 발견 문제 모두 hotfix 또는 알려진 이슈 등록

## 확인 질문
- 학교 PC가 ARM64인 경우 — ARM64 NSIS 빌드 사용. 서명되지 않은 ARM64 인스톨러가 SmartScreen에 차단될 가능성. IT 담당자와 사전 조율.
- 학교 네트워크에서 외부 API 차단 — Anthropic/OpenAI/OpenRouter 도메인 화이트리스트 요청 필요. 사전에 IT 담당자에게 요청 목록 전달.
- 학생 노트북도 사용한다면 — 다양한 OS·사양 가능성. 25명 모두 동일 환경은 아니므로 최소 사양·테스트 가이드 작성.
- 시뮬레이션이 비용 발생 — OpenRouter 자식 키 25개 × 50회 호출. 토큰 한도 작게 설정해 비용 최소화.
