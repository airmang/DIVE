# 작업 32: 접근성 (키보드·ARIA·스크린리더)

## 컨텍스트
명세 §12.2에 따른 접근성 보강. 모든 액션은 키보드만으로 수행 가능해야 하고, 스크린 리더 사용자도 진행 상황을 파악할 수 있어야 함.

## 이번 작업 범위
- 전체 키보드 단축키 정의·구현 (명세 §12.2)
- 모든 인터랙티브 요소에 ARIA 라벨
- 상태 변경 시 `aria-live` 영역으로 알림
- 워크맵 카드 키보드 탐색 (Tab/Shift+Tab/Enter)
- 권한 카드 키보드 단축키 (A/R/E)
- 색상 대비 검증은 작업 33에서

## 명세 참조
- DIVE_SPEC.md §12.2 — 접근성

## 단계

1. **키보드 단축키 시스템**:
   - `src/lib/shortcuts.ts` — 단축키 등록 추상화 (`react-hotkeys-hook` 또는 자체)
   - 명세 §12.2 단축키 모두 구현:
     - `Ctrl+N` — 새 프로젝트
     - `Ctrl+S` — 체크포인트
     - `Ctrl+/` — 프롬프트 도우미
     - `Ctrl+,` — 설정
     - `Ctrl+E` — 슬라이드 인 패널 토글
     - `Ctrl+W` — 워크맵 펼침/접힘
   - 단축키 도움말 — `Ctrl+?` 또는 메뉴
2. **워크맵 카드 키보드 탐색**:
   - Tab/Shift+Tab으로 카드 좌우 이동
   - Enter로 진입
   - 카드 내부에서도 Tab 가능 (액션 버튼 등)
   - 가로 스크롤 자동 따라가기 (현재 포커스 카드가 화면 안에 보이도록)
3. **권한 카드 키보드**:
   - 카드 활성 시 자동 포커스
   - `A` — 승인, `R` — 거부, `E` — 편집
   - Tab으로 [승인]/[수정]/[거부] 버튼 순환
4. **ARIA 라벨**:
   - 모든 아이콘 전용 버튼 — `aria-label`
   - 워크맵 카드 — `aria-label="카드: {title}, 상태: {state}"`
   - 권한 카드 — `aria-label="도구 호출 승인 요청: {tool}"`
   - 토글 버튼 — `aria-pressed`
   - 모달 — `role="dialog"`, `aria-labelledby`, `aria-describedby`
5. **aria-live 영역**:
   - 채팅 영역 — `aria-live="polite"` (새 메시지 알림)
   - 토스트 — `aria-live="assertive"` (즉시 알림)
   - 워크맵 카드 상태 변경 — `aria-live="polite"`
   - 권한 카드 등장 — `aria-live="assertive"` (사용자 결정 필요)
6. **포커스 관리**:
   - 모달 열림 시 첫 입력 요소에 포커스
   - 모달 닫힘 시 호출 요소로 포커스 복귀
   - Focus trap — 모달 안에서 Tab 순환
7. **스크린 리더 테스트**:
   - Windows Narrator로 핵심 시나리오 동작 확인
   - 시나리오 A 1~6번을 스크린 리더로만 따라가기
8. **단위 테스트** — `@testing-library/jest-dom`의 `toHaveAccessibleName` 등 검증

## 확인 질문
- 스크린 리더 테스트 — Narrator(Windows 기본) vs NVDA(무료, 더 전문). 둘 다 추천. 학교 환경에서는 Narrator 가능성 높음.
- 단축키 충돌 — 운영체제 단축키와 충돌 가능 (Ctrl+W = 창 닫기). 다른 키로 변경? 또는 사용자가 설정에서 변경?
- 모달 포커스 trap — `react-focus-lock` 라이브러리 vs 자체 구현. 라이브러리 추천.
- aria-live 빈도 — 매 메시지마다 알림은 과한가? 메시지 묶음(예: AI 응답 1개 단위)으로? 묶음 추천 (스크린 리더 부하 감소).
