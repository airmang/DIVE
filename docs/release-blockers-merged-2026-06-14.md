# DIVE 출시 차단 — 정적+동적 통합 진실 (2026-06-14)

정적 감사(`release-audit-2026-06-14.md`, 코드 7차원) + 동적 감사(`dive-release-dynamic-audit-2026-06-14.md`, Codex computer-use 실구동)를 교차 대조해 하나로 정리.

## 핵심 해석 주의 — 좌표 아티팩트 오염

동적 감사의 **D-33**(보이는 버튼과 실제 클릭 좌표가 수십~100px 어긋남)은 **DIVE 버그가 아니라 computer-use의 Retina/스케일링 아티팩트일 가능성이 높다**(스크린샷 2x ↔ 클릭 논리좌표 1x 불일치는 computer-use의 흔한 문제). 만약 그렇다면, *정밀 클릭에 의존하는* 다수의 "죽은 버튼" 동적 발견(D-07, D-16, D-17, 일부 A4, D-25)은 **실제 죽은 핸들러가 아니라 빗나간 클릭**일 수 있다.

→ **규칙: 동적 "죽은 버튼"은 정적 코드로 교차검증한 뒤에만 수정한다.** 핸들러가 실재+동작하면 좌표 미스(아티팩트), 핸들러가 부재하면 진짜 dead-wiring. (단 D-08·D-13처럼 정적이 이미 dead로 확정한 건 진짜다.)

## 종합 판정

**출시 불가 — 그리고 "폴리시 단계"가 아니다.** 동적이 정적은 못 본 **세 구조적 문제**를 잡았다: ① 핵심 버튼이 앱을 통째로 크래시, ② 프로젝트/세션/계획-draft **상태 모델이 불안정**(draft 소실·화면 불일치), ③ 좌표/히트영역 의심. 특히 ②는 *감사 자체를 막아* keystone 테스트(A1/A3)에 도달조차 못 하게 했다. **핵심 흐름이 아직 안정적이지 않다 = 중반 단계.** 내가 만든 provocation P0~P4는 이 불안정한 토대 위의 폴리시였다.

## 권장 순서 — "핵심 토대 먼저, 폴리시 나중"

1. **B2-CRASH** — 크래시는 무조건 1순위. (크래시 로그 확보 → 프리뷰 렌더/IPC 패닉 추적)
2. **B2-STATE** — 상태 모델 안정화. *이게 안 되면 나머지 검증·데모·수정 전부가 불안정.* 아키텍처 조사 필요.
3. **Bucket 3 disambiguate** — 좌표 의심 발견을 코드로 분류(아티팩트 vs 진짜). 빗나간 클릭 쫓느라 시간 낭비 방지.
4. **B1 dead-wiring 근본수정** — 5개 host 분산 switch를 중앙 resolver 하나로(B1-a/b/c/d 동시 해소). 패치 아님.
5. **B1-g keystone** + **B1-e i18n**(슬라이드인·메뉴).
6. **B1-h export 유출 + B1-i guard + B2-DEL 삭제확인** — 보안·데이터.
7. **B1-f 복원 a11y + 나머지 B2/P2**.

**메시지**: 크래시 + 불안정한 상태 모델이 토대에 있다. 카드/검증 폴리시는 그 위에 있어 토대가 흔들리면 같이 무너진다. 안정화가 먼저다 — 그래야 keystone도 제대로 시연된다.
