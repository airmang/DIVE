# DIVE — E2E Journey Gap Backlog

From 50 real-user E2E journeys (design→done) gap-analyzed against the codebase. **127 gaps** found. Engine: multi-agent Workflow (21 agents, code-grounded). Journeys in [e2e-journeys.md](e2e-journeys.md).

**Severity:** P0 22 · P1 57 · P2 48  |  **Confidence:** high 70 · med 47 · low 10  |  **By stage:** verify 67 · implement 17 · recover 14 · design 11 · cross 10 · plan 8

> The verify stage holds **half the gaps** — DIVE's core differentiator (evidence-gated supervision) is also where it most often falls short.

## Prioritized themes (→ spec-kit specs + wily Stages)

### 1. [P0] i18n localization debt (Korean leaks at English locale)  ·  effort L  ·  24 gaps
The single most pervasive defect class (DEF-01 at scale). Two sub-roots: (1) frontend pure-modules hardcode Korean string literals never routed through t() — agencyStatus.ts AGENCY_STATE_META (12-13 labels), verificationStatus.ts evidence chips, PlanAddStepPanel SCOPE_ACTION_ROUTE_COPY, CheckpointTimeline labels, useChatSession error strings; (2) backend prompts/messages are Korean-only or force Korean output regardless of locale — supervisor.rs:1426 'Ask one criterion-linked Korean question', verify.rs Korean verdict prompt+schema, verification_coach.rs Korean unavailable_response + no output-language directive, assist.rs Korean D-stage prompt, agent/mod.rs locale_system_prompt written in Korean, ipc/preview.rs + ipc/cards.rs auto_checkpoint_label Korean. Every English persona sees Korean in trust-critical surfaces: provocation card questions, agency/evidence badges, checkpoint restore points, coach guidance. This is the #1 thing breaking a credible English E2E experience.

- **Proposed spec:** Spec: locale-correct surfaces end-to-end. (1) Frontend: replace hardcoded Hangul literals in agencyStatus.ts, verificationStatus.ts, PlanAddStepPanel, CheckpointTimeline, useChatSession with i18n keys resolved via t()/useT() at render; add en/ko entries; add an ESLint no-literal-Hangul guard (excluding i18n catalogs) + a snapshot test asserting the en render contains no Hangul. (2) Backend: thread context/request.locale into build_supervisor_prompt, verify.rs build_system_prompt+schema, verification_coach.rs prompt+unavailable_response, assist.rs, agent/mod.rs locale_system_prompt, preview.rs logs/errors, cards.rs checkpoint labels — branch on locale_is_english (mirror plan_interview.rs) so the question/verdict/guidance/labels are emitted in the active UI locale. Add a test asserting the supervisor prompt contains 'English' for an en locale.
- **Affected journeys:** STATIC-02, STATIC-04, INTERACTIVE-01, INTERACTIVE-02, INTERACTIVE-03, INTERACTIVE-05, CROSS, STYLED-03, DATA-cross, APPS-cross, REFACTOR-02, REFACTOR-03, REFACTOR-04, REFACTOR-05, DEBUG-01..05, ADDFEATURE-05 …

### 2. [P0] Verification evidence is hollow — typing/clicking clears the gate  ·  effort L  ·  18 gaps
DIVE's core anti-automation-bias promise leaks. manual_observation counts as concrete from any non-empty text (canRecord just needs trim().length>0), and acceptanceCriterionConfirmed is a single global boolean covering ALL criteria, so one keystroke or one tab-switch clears the decision gate. Specific failures: print/PDF/responsive/dark-mode/reduced-motion ACs cannot be machine-verified yet pass on free text; the parallel manual-observation path bypasses the correctly-gated preview-confirm button (FR-030~033 click≠evidence leak); multi-criterion fixes (search case-insensitive/partial/empty, regression re-test, money math, pagination boundaries) approve after observing only one criterion; preview-confirm enables on a tab-switch click without iframe onLoad/ready or any interaction signal.

- **Proposed spec:** Spec: criterion-scoped, substantive evidence gate. (1) Track evidence per acceptance_criterion id (reuse verification_coach criterion_ids); gate canApproveDirectly on coverage of ALL gating criteria, not one global boolean — surface 'N of M criteria observed' as a blocking reason. (2) Require manual_observation to echo a concrete value/action (enum action for print/PDF; min substantive length; reference the criterion value) before it counts as concrete. (3) Close the parallel free-text loophole: when a runnable/openable check is feasible, require the open/click/test action before manual_observation satisfies that criterion. (4) Gate preview-confirm on previewSession.status==='ready' (iframe onLoad) plus a typed criterion-linked observation, not a tab-switch.
- **Affected journeys:** STATIC-01, STATIC-03, STATIC-04, STATIC-05, INTERACTIVE-03, CROSS, STYLED-05, DATA-04, DATA-05, APPS-01, APPS-02, DEBUG-03, DEBUG-04, ADDFEATURE-05, CAREFUL-01

### 3. [P0] In-app preview cannot exercise the behavior under test  ·  effort XL  ·  17 gaps
The verify step — the reason DIVE exists — pushes users out of the supervised loop precisely where it matters. The preview iframe is fixed h-full w-full in a 520px panel with NO width/device control (responsive breakpoints, column-count changes unobservable), no reload button (persistence/localStorage survival unprovable; re-clicking Open keeps same src so no reload), no color-scheme/reduced-motion emulation, and a sandbox missing allow-modals/allow-forms (confirm()/alert()/<form> submit suppressed — ADDFEATURE-01's confirm-before-clear literally cannot be shown). Two window-level Esc listeners close the panel when the user tests close-on-Esc. The iframe console/network is not surfaced, so empty-vs-broken and JS errors are invisible. CSP connect-src blocks third-party API fetch and even local ./products.json from the asset: origin, so data-fetch journeys render empty for reasons the agent's self-report won't catch.

- **Proposed spec:** Spec: a real, observable preview. (1) Responsive toolbar: preset width buttons (mobile 375 / tablet 768 / desktop) + draggable handle + width readout; each width selection emits a distinct preview observation tied to the breakpoint AC. (2) Reload button (bump iframe key / contentWindow.location.reload()) emitting a 'reloaded' observation for persistence ACs. (3) Sandbox: add allow-modals and allow-forms (keep top-nav/popups excluded). (4) Scope Esc-to-close handlers to ignore Escape when focus is in the preview iframe. (5) Serve static previews over loopback http (already partly supported) and widen/relax CSP connect-src for static previews so fetch works; surface a 'network blocked' banner + iframe console/onerror stream into TerminalTab when it doesn't. (6) Optional color-scheme/reduced-motion emulation toggles; warn in the verify card when a diff uses prefers-* the preview cannot prove.
- **Affected journeys:** STATIC-01, STATIC-02, INTERACTIVE-01, CROSS, STYLED-01, STYLED-02, DATA-01, DATA-02, DATA-04, APPS-01, APPS-05, DEBUG-01, DEBUG-02, ADDFEATURE-01, ADDFEATURE-02, ADDFEATURE-03 …

### 4. [P1] Recovery: file-only restore, no pre-edit/pre-pivot/pre-DIVE anchors  ·  effort L  ·  11 gaps
Undo — the skeptical user's one dependable feature — is half-built. restore_checkpoint reverts the git worktree only, NOT the SQLite card/chat/plan/roadmap state, producing a confusing half-reverted project (CAREFUL-03). Checkpoints are created only on card transitions, not before each approved write, before a plan pivot, or from pre-DIVE git history — so a broken mid-step file or a project broken before opening DIVE (DEBUG-05) has no clean restore point. The retry-loop/STOP signal only fires on >=2 IDENTICAL normalized errors from verify-stage failures, so slightly-varying errors or implement-stage conversational fix→break loops on commandless static projects never trip it. Restore is reachable only from a top-bar control, never surfaced at the moment a fix fails. The TopBar badge count (recoveryCount + failed-step) disagrees with the panel's checkpoint list.

- **Proposed spec:** Spec: trustworthy recovery. (1) On restore, also roll back card/plan/roadmap DB rows (snapshot a session-state blob alongside git_sha; reapply in one transaction) OR clearly warn that restore reverts files only. (2) Auto-checkpoint before each approved write, before plan pivots/replan, and surface a 'restore point created' affordance. (3) On project open of a git repo, expose existing git history as restore points (or warn only post-open states are recoverable). (4) Add a per-step total-failure counter (fires retry-loop at N>=3 regardless of fingerprint) and feed conversational retry signals + implement-stage failures into it; coarsen the fingerprint. (5) Surface an inline 'Restore to last checkpoint' at failure/retry-loop time. (6) Reconcile TopBar badge count with the panel list.
- **Affected journeys:** STATIC-01, INTERACTIVE-05, REFACTOR-01, DEBUG-05, CAREFUL-03, CAREFUL-05, FRICTION-01

### 5. [P1] Plan-edit routing: add-only, no clarify/remove/multi-step  ·  effort L  ·  7 gaps
The plan router (plan_router.rs / workspace_plan.rs) only ever returns AddStep or Chat — there is no clarifier turn, no remove/supersede/modify step, and no multi-step fan-out. Consequences: ambiguous adds draft criteria from one LLM line with no human clarification (no 'valid email = ?', no 'Clear on empty list?'); a mid-plan pivot bolts on new steps while obsolete approved steps linger (FRICTION-01); a genuinely multi-part ask (new page + nav on both pages + link) is crammed into one oversized step or silently dropped to chat (ADDFEATURE-04); find_duplicate_step can silently downgrade a real add to chat leaving no visible outcome; and the RationaleChallengePanel is a deterministic Korean stub that never engages the user's objection.

- **Proposed spec:** Spec: full plan-mutation routing. Add router outcomes/IPCs for ROUTE clarify (one criterion-linked question before drafting, answer feeds the draft), remove/supersede_step (keep in history, hide from active plan), and an ordered multi-step fan-out for larger asks presented as a reviewable plan-diff (+N new / ~M changed / kept). On duplicate detection surface a confirmation ('similar to step X — add anyway / open X') instead of silent downgrade. Route the rationale-challenge through the SupervisorAgent for a real accept/counter/adjust response.
- **Affected journeys:** ADDFEATURE-01, ADDFEATURE-02, ADDFEATURE-04, FRICTION-01, FRICTION-04

### 6. [P1] Beginner approval-card legibility (no plain-language change summary, no read gate, no secret/scope warnings)  ·  effort L  ·  6 gaps
For non-technical owners and zero-code beginners the write/edit approval card assumes diff-literacy: the only explanation is a static per-tool i18n string ('writes content to a file', 'edits a file'), never 'adds your opening-hours table' or 'this :hover rule changes the background'. WarnCard exposes an approvalRequirement read-gate but NO caller ever passes it, so Approve is a single frictionless click — the rubber-stamp condition. There is no per-line/per-change teaching for learners, no secret/.env detection on the actual key-writing file path (only shell commands are scanned), and a whole-file write_file overwrite of an existing page shows a large noisy diff with no 'unrelated region changed / N lines removed' highlight.

- **Proposed spec:** Spec: legible, gated approval. (1) Generate a short per-change natural-language summary from the patch/args ('adds an opening-hours table with 7 rows'; an optional 'explain this change' affordance for learners) rendered above the diff for content writes. (2) Wire WarnCard.approvalRequirement for write/edit: require the diff to be expanded/scrolled (diffViewed flag per toolCallId) or an 'I read the change' confirm before Approve enables; keep read_file auto-approved. (3) Add secret/.env heuristic to the write/edit pipeline with a distinct WarnCard callout + risk escalation. (4) Prefer edit_file over whole-file write_file and warn 'this replaces the entire file (N lines removed)'.
- **Affected journeys:** STATIC-03, STATIC-04, INTERACTIVE-01, INTERACTIVE-04, DATA-03, APPS-05, ADDFEATURE-04

### 7. [P1] PRD/Socratic interview friction and weak criterion scaffolding  ·  effort L  ·  9 gaps
The interview is the first thing every beginner hits and can cause bail-before-value: it demands full concreteness on 5 dimensions + 2 ACs over 3-6 exchanges with no progress hint, no skeleton/momentum preview, and no express/batch-intake fast path for trivially-scoped static pages. Criterion QUALITY is left entirely to the LLM — nothing deterministically scaffolds responsive/persistence/a11y criteria for UI goals or loading/empty/error criteria for fetch goals, and nothing enforces that 'ready' criteria are actually concrete/independently-checkable (required only checks presence). Vague-answer detection and ambiguity hinting are Korean-regex-only, so English vague input ('just make it nice') gets zero hints. A static index.html has no runnable verification_command, so the schema pushes a bogus/null command with no manual/preview enum to route static projects cleanly.

- **Proposed spec:** Spec: faster, higher-quality interview. (1) Progress hint ('2 more quick questions') + instant skeleton preview once the goal is captured; optional structured quick-intake form for simple static categories that satisfies PRD in one submit, falling back to Socratic only when answers are vague. (2) Deterministic post-draft checks: UI goals must carry responsive/persistence/a11y criteria; fetch goals must carry loading/empty/error criteria; each acceptance_criterion must be independently checkable — list missing ones in unresolved_questions and route back rather than presenting a 'ready' PRD. (3) Add English ambiguity/vague rules (locale-keyed) or move detection into the LLM interview. (4) Add a verification_type enum (run|preview|manual|test) so static front-end steps route to preview/manual with no command.
- **Affected journeys:** STATIC-01, STATIC-05, INTERACTIVE-02, STYLED-03, DATA-cross, APPS-02, ADDFEATURE-03, FRICTION-03

### 8. [P1] Verification stepper false-completion & detection robustness  ·  effort M  ·  8 gaps
The 4-stage stepper (코드 이해→점검·관찰→검토 응답→결정) marks a stage 'completed' with a green checkmark simply by navigating away — visitStage adds to completedStageIds on any move with no evidence precondition. The real gate is the decoupled DecisionGate, so beginners get green checks for clicking through nothing, teaching that clicking == verification and confusing them about why they're still blocked. Adjacent robustness gaps: AiClaimedDone completion detection is a closed phrase list (thin English coverage — misses 'all set', 'your button works again'); retry/recovery intent detection is Korean-keyword-biased; feature/vague-goal provocation rules (rules.ts) are Korean-keyword-biased so English input under-triggers; and non-P1 supervisor events (PlanDrafted/DiffReady/ScopeExpansion/RetryLoop) silently vanish offline with no deterministic shell fallback.

- **Proposed spec:** Spec: honest stepper + locale-robust detection. (1) Tie a stage's checkmark to its own evidence (점검·관찰 checks only once an observation/preview-open recorded; 결정 only once a decision is made); render 'visited' (neutral) vs 'evidenced' distinctly, or gate Next on evidence. (2) Trigger the verify provocation on VerifyEntered/stage-entry independent of phrase matching; broaden English completion/retry/feature/vague term lists or move to structured backend signals. (3) Provide deterministic, locale-aware shell decisions for plan/diff/scope/retry events so provocations degrade gracefully offline.
- **Affected journeys:** STATIC-04, CROSS, DEBUG-02, DEBUG-cross, REFACTOR-02, ADDFEATURE-03, CAREFUL-04

### 9. [P1] Production scope-drift / high-risk-file gate is dead code  ·  effort M  ·  2 gaps
The DecisionGate's high_risk_unexpected_files reason can never fire in a shipped build. The gate reads flat metadata.highRiskFiles from cards with severity==='risk', but the live SupervisorAgent hardcodes severity to the single Caution variant and buries the list under metadata.assessmentSummary.highRiskFiles; the only cards that emit severity='risk' are dev-quarantined behind import.meta.env.DEV. So Daniel's expectation — touching package.json off-target forces a written risk reason even when the feature works — silently never happens, and a supervisor parse-failure leaves drift surfacing nowhere.

- **Proposed spec:** Spec: deterministic high-risk drift backstop. Make the gate read metadata.assessmentSummary.highRiskFiles (or flatten high_risk_files into metadata.highRiskFiles in the adapter) and treat a populated high-risk list as gate-blocking regardless of card severity; emit the blocking signal directly from diff_ready_assessment.high_risk_files independent of whether the LLM card parses, so scope creep onto high-risk paths is caught even when the supervisor degrades.
- **Affected journeys:** CAREFUL-04

### 10. [P2] Static-vs-server preview discoverability & onboarding  ·  effort M  ·  4 gaps
For the exact single-index.html beginner persona the category targets, the big primary Preview button (autoConnect) is hard-wired to the dev_server path and fails with a hardcoded Korean 'select a web project with package.json' error; the static index.html action is a secondary outline button. The empty-state copy wrongly says a dev server must be running. For Tailwind/React projects the opposite is true — a raw static file makes responsive classes silently no-op. There is no first-run onboarding/coachmark explaining static-vs-server preview or that the human (not AI) must confirm the result.

- **Proposed spec:** Spec: correct preview defaults + orientation. Change autoConnect to openPreview('auto') (backend Auto already prefers static index.html); make detect_package_info return an i18n code not raw Korean; rewrite empty_title/empty_description to lead with the static-file option and surface the index.html candidate as the primary affordance when root/index.html exists; steer package.json/Tailwind projects to dev-server with install/build progress + clear failure. Add a one-time Preview/verify coachmark on static-vs-server and human-confirms-result.
- **Affected journeys:** INTERACTIVE-01, STYLED-02

### 11. [P2] Refactor/rename safety: no behavior-preserving guidance, no multi-replace, noisy search  ·  effort M  ·  4 gaps
Refactor journeys lack guardrails. No agent/supervisor prompt frames a step as behavior-preserving ('move verbatim, do not alter logic'), so the LLM's rewrite-while-extracting tendency is caught only by a human reading diffs; and no minimal-fix/diagnose-before-edit guidance exists for debug journeys, making a broad rewrite as likely as a one-line fix. edit_file fails on >1 match (good for safety) but with no rename/multi-replace tool a common-identifier rename becomes N approval cards — a usability cliff for beginners. search_files caps at 100 and doesn't exclude node_modules/dist, so 'found N usages' completeness evidence can be polluted or truncated.

- **Proposed spec:** Spec: safe refactor primitives. (1) When the PRD marks a step behavior-preserving/refactor/comment-only, inject an agent instruction to move code verbatim and avoid logic edits, and show a 'behavior-preserving step' banner on the approval card; add a minimal-fix/diagnose-before-edit instruction for debug steps. (2) Add a scoped multi-replace/rename tool (or replace_all with an N-occurrence count preview on one approval card). (3) Exclude node_modules/dist/build/.git from search_files, raise/paginate the cap, and surface 'scanned X, capped'.
- **Affected journeys:** REFACTOR-01, REFACTOR-02, REFACTOR-03, DEBUG-cross

### 12. [P2] Coach degradation & criterion-specific fallback guidance  ·  effort M  ·  5 gaps
Verification-coach degradation is well-built (every failure maps to a non-hanging Unavailable state, observation recording is independent) — but the fallback is a single generic 'guidance unavailable' line instead of criterion-keyed manual steps, so when the LLM guide drops, a beginner (a11y/keyboard-nav, etc.) gets an empty observation box with no how-to. The coach also can't ask data-type-specific questions (numeric-vs-lexical sort) because it only receives changed-file names, not diff/comparator context. No retry/backoff is confirmed for a rate-limited provider key (retry-storm risk).

- **Proposed spec:** Spec: useful degraded coach. On unavailable, fall back to deterministic per-criterion checklists derived from the step's acceptance_criteria (ship templates for keyboard-nav, label association, contrast, behavior-unchanged, sort-type) instead of a generic line. Feed a bounded diff/comparator snippet into the coach context so it can ask data-type-specific questions. Confirm providers/retry.rs honors 429/Retry-After with capped backoff and surface a cooldown countdown in the ProviderSetupBanner.
- **Affected journeys:** APPS-05, DATA-02, REFACTOR-04, DEBUG-04, FRICTION-05

## Synthesis notes

Verified against source in ~/DIVE-2/dive: supervisor.rs:1426 hardcodes the Korean provocation question; PreviewTab.tsx:247 sandbox is 'allow-scripts allow-same-origin' (no allow-modals/allow-forms); agencyStatus.ts:28+ hardcodes Korean labels. These three confirmed roots underpin the top 3 themes.

Sequencing logic: the top 3 themes are P0 because each independently breaks a credible English E2E run — i18n makes trust-critical surfaces unreadable, the hollow evidence gate defeats DIVE's reason-to-exist, and the inert preview pushes users out of the supervised loop exactly at verify. They also share fixes: a responsive/reload/ready-gated preview (theme 3) is the mechanism that lets the evidence gate (theme 2) demand real per-criterion observation, so build them together. The supervisor.rs locale fix is shared between theme 1 and the stepper/detection theme.

Effort note: theme 3 (preview) is XL because sandbox flags are S but loopback-serving + CSP relaxation + console bridging + per-breakpoint observation plumbing compound. Themes 1, 2, 5, 6, 7 are L mostly due to breadth (24/18/11/6/9 gaps) not individual difficulty — many constituent fixes are S/M and can land incrementally.

Two confirmed NON-gaps worth preserving as regression guards (do not 'fix'): verification-coach sidecar unavailability degrades cleanly without hanging (APPS-05/FRICTION-05), and PlanDraftRecoveryScreen retry preserves persisted interview answers and is i18n-clean (FRICTION-05) — add integration tests to lock both.

Did not Read every file end-to-end; prioritization relies on the provided per-gap evidence plus spot-verification of the three highest-frequency P0/P1 claims. Confidence in theme clustering and ordering is high; individual effort sizes are directional.
