# DIVE — Overnight Live-Run Queue (11 full E2E journeys)

Sequential unattended run. Each iteration: pick the next `pending` journey → create a fresh sandbox project `qa-sandbox/live-<id>/` → set the persona locale → drive FULL design→PRD→plan→implement→verify→done on the real `.app` via computer-use → record gaps to [live-run-results.md](live-run-results.md) → mark `done` (or `blocked: <stage/reason>` if stuck) here. caffeinate prevents sleep. On a stuck state: retry button clicks once, try keyboard, then log `blocked` and move on — never halt the whole run.

**Driving notes:** action-button clicks can intermittently not fire → retry / use Cmd+Return for chat sends / answer the Korean approve-gate `없음 — 승인 가능`. PRD board only fills from an explicitly structured `Goal:/Done criteria:(1)(2)/Out of scope:` message. Plan JSON may fail first try → use Regenerate short JSON. Preview defaults to dev-server (unavailable) → click the `index.html` chip. Take a fresh screenshot before each click.

## Per-journey specs


### STATIC-05  (Static web pages (beginner))  ·  locale=ko  ·  beginner
- **Persona:** Jay, a high-school club president (teen, allowed solo use — no minor-pilot/IRB involvement, just an individual user) making a one-page info site for the school coding club: what the club does, meeting times, how to join, and a few photos. Impatient, wants it fast, Korean speaker.
- **Goal:** A single info page for the coding club: hero with club name + slogan, an 'about' paragraph, a meeting schedule, a 'how to join' section with a Google Form link, and 3 event photos.
- **Design:** Jay wants to skip straight to building and is mildly annoyed the Socratic interview appears first. But the questions are concrete (slogan? meeting day/time? the Google Form URL? which photos?), so he answers fast. The board captures scope and ACs ('join section links to the club's Google Form', 'schedule lists weekly meeting', 'three photos in a row'). He confirms the PRD, slightly impatiently.
- **Implement/Verify focus:** He blasts through steps, approving each write diff fast (impatience = high automation-bias risk). The agent reads the photos dir; the diff shows the <img> wiring. He opens Preview and the club page renders — he's hyped. He pastes the Google Form URL into chat and the agent wires the join button.  //  Agent claims done. A provocation card asks the criterion-linked question: 'The join button is supposed to open your Google Form — did you click it and confirm the URL, or is that AI self-report?'. The verify stepper blocks 결정 until he provides evidence. Initially he tries to skip, but the stepper won't advance without him opening the Preview and actually clicking the join link / checking the href. He finds the agent slightly mangled the long Form URL, fixes it via a corrective edit, re-checks, then verifies. This is the journey's whole point — supervision catches the one thing he would have shipped broken.
- **Repro target (why-live):** P0 click≠evidence leak: try to clear the mangled-URL verify via one keystroke in the free-text manual-observation field while never opening the preview — directly tests the core anti-rubber-stamp promise.
- **Watch for:** This persona is the stress test for automation bias: an impatient teen will click-approve every diff and try to click-through verify. If the verify stepper ever accepts a click on the card (not real evidence) as sufficient, the supervision collapses exactly for the user who needs it most — the 'click ≠ evidence' (FR-030~033) principle must hold here. The mangled long-URL is a realistic LLM failure the app can't detect on its own; the whole value hinges on the card forcing a human click-test. Korean LLM replies are fine for him. Risk: if the PRD interview feels slow, an impatient user bails before reaching the part where supervision adds value; the interview must front-load the concrete questions and feel fast.

### ADDFEATURE-01  (Add a feature to an existing project)  ·  locale=en  ·  beginner
- **Persona:** Mina, a high-school student who built a small personal to-do web page in DIVE last week (one index.html with a list and an 'Add' input). She knows almost no JavaScript and thinks in terms of 'I want it to do X', not code.
- **Goal:** Add a 'Clear all' button to her existing to-do list page that removes every task at once, with a confirm so she doesn't wipe the list by accident.
- **Design:** Mina reopens her existing to-do project from the sidebar; the prior PRD (v1), plan, and index.html are already loaded. She types in chat, in plain language: 'add a button that clears all the tasks, but ask me to confirm first.' She does NOT expect a brand-new PRD interview from scratch — she expects DIVE to recognize this as an addition to the project she already has. What she should see: the plan router catches the request and offers to fold it in as a new step rather than treating it as idle chat, plus a short clarifier (e.g. 'should Clear all also work when the list is already empty?') that links to her existing acceptance criteria.
- **Implement/Verify focus:** The agent loop runs the step. A read of index.html auto-approves (safe tier, no card). The write/edit to add the button and its click handler surfaces a PERM card with a diff preview (+the button markup, +a confirm()/handler block). Mina reads the diff, sees it only touches her page near the existing input, and clicks 'allow this change'. She expects exactly one approval prompt for the actual edit, with a readable before/after.  //  A verification stepper opens (코드 이해 → 점검·관찰 → 검토 응답 → 결정). A criterion-linked provocation card asks her to prove it actually works, not just that code was written: 'Did the list actually empty when you clicked Clear all — and did the confirm appear?' She opens the preview tab, runs her page, adds two tasks, clicks Clear all, sees the confirm, accepts, watches the list empty. Opening the diff flips '코드 이해' to Diff 확인됨; her live observation records the behavior. She then records the decision. The card is non-blocking but she's nudged to confirm with real evidence.
- **Repro target (why-live):** P0 sandbox-modals: confirm-before-clear feature whose window.confirm() is suppressed by the iframe sandbox — verification literally cannot show the one behavior the card demands.
- **Watch for:** (1) The router mis-classifying a clear feature add as plain 'chat' so nothing gets added to the plan, leaving Mina confused about whether it worked. (2) The agent scope-creeping (restyling the whole page, adding unrelated features) so the diff is bigger than 'one button + confirm'. (3) The confirm dialog being a native browser confirm() that the preview iframe may suppress, so verification 'looks' done but she never actually saw the confirm. (4) DEF-01 / Korean leakage: if her locale is English she may see Korean status badges (검증 필요, 계획 검토 필요) on the new step, and the LLM may answer her in Korean.

### STYLED-02  (Styled / responsive UI)  ·  locale=en  ·  intermediate
- **Persona:** Jordan, a bootcamp grad building his first freelance landing page. Comfortable with React/Tailwind concepts but new to agentic tools. His desktop nav bar overflows on phones and he needs a hamburger menu.

- **Implement/Verify focus:** The agent reads his component and CSS (auto-approved), then proposes write/edit cards for the markup and the media-query/breakpoint styles. The core acceptance criterion is responsiveness at a specific width, but DIVE's Preview tab iframe has no width/device control — so Jordan opens the preview, sees it at panel width, then realizes he must resize the DIVE window (or open the page in a real browser's responsive mode) to actually cross the 768px breakpoint. He likely has to leave DIVE for a real browser, breaking the in-app supervised loop. Also watch for Tailwind not being configured/built in a static preview (responsive classes silently no-op), and Korean LLM replies/review-card copy under English locale (DEF-01).
