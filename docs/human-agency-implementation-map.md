## Implementation Boundary

DIVE should implement Human Agency / Black Box visibility by extending the existing Decompose, Instruct, Execute, Verify, Extend workflow. The default implementation path is not a new `src/features/agency/*` workflow, a standalone dashboard, a quiz, a card deck, a worksheet, a score, or a badge system.

If later phases need helper code, prefer adapter names that sit over existing systems, such as `dive/src/features/provocation/agencyMapping.ts` or `dive/src/features/roadmap/agencyStatus.ts`. Keep `CardState`, `RoadmapStepStatus`, `StepSessionMapping`, checkpoint state, and EventLog/export rows as the durable workflow backbone.

## Duplicate UI Risks

Avoid these implementation paths:

- A standalone "Human Agency" dashboard or screen. Use Roadmap, Step detail, PermissionCard, RecoveryPanel, and export docs.
- A second Action Card under `PermissionCard`. `PermissionCard` plus contextual review card host is the action surface.
- A second Review Card system. Extend `dive/src/features/provocation/*` and keep `검토 카드` / `확인 필요 카드` terminology.
- Chat-message warnings for generic AI safety. Cards must be grounded in concrete goal, plan, diff, tool, verify, checkpoint, or export state.
- Treating `verify_log.intent_match`, assistant completion text, or diff-opened status as `verified_with_evidence`.
- Long reflection gates for low-risk approvals. Keep Work Mode low-friction.
- Persisting raw prompt, source, terminal body, secret, or env contents in EventLog/export. Use current summary/anonymizer patterns.

## Priority Order From Audit

1. Phase 1: create a derived agency-state adapter over existing provocation, roadmap, workmap, checkpoint, and verification data.
2. Phase 5: add the embedded Decision Gate because it defines how verify, diff, rollback, and risk reasons are interpreted.
3. Phase 4: merge changed files, high-risk drift, diff reviewed, verification chips, decision state, and rollback availability in Step detail.
4. Phase 3: normalize pending action context and preserve high-risk approval metadata through EventLog/export.
5. Phase 6: make rollback/repro/scope-reduction precede another AI retry in failure and regeneration-loop states.
6. Phase 7: add agency-derived EventLog/export metadata without duplicating every event type.
7. Phase 2: refine Intent/Plan UI copy and action wiring after state and decision semantics are stable.
8. Phase 8: finish copy, settings, and mode-policy tests.
9. Phase 9 and Phase 10: run manual scenarios and full verification commands.

- `provocation.card_shown`, `provocation.action_clicked`, `provocation.dismissed`, `provocation.marked_irrelevant`, `provocation.continued_with_risk` through `dive/src/features/provocation/logging.ts` and `dive/src-tauri/src/ipc/provocation.rs`.

- `card_update` with approval provenance summaries in `dive/src-tauri/src/ipc/cards.rs`.
- `checkpoint_create` and `checkpoint_restore` in `dive/src-tauri/src/ipc/checkpoint.rs`.
- Card export fields: `approval_judgment`, `approval_judgment_metrics`, `approval_provenance`, `verification_evidence_summary`, `changed_files`, `verify_log`.
- Step mapping export fields: `verification_status`, `verification_evidence`, `user_decision`, `status`, `completed_at`.
- Event export sanitization for provocation payloads in `dive/src-tauri/src/export/mod.rs`.

## Events And Export Fields Still Missing

- A richer `affectedCommands` field for future non-shell tool families if they expose structured command metadata.
- Optional durable cross-session diff-viewed event if later research needs to reconstruct UI-open state beyond the current session/provenance.

## Phase 0 Conclusion

Current HEAD already has enough typed models, deterministic rules, provenance fields, and export paths to start Phase 1 without creating a parallel agency workflow. The safest next implementation step is a small derived agency-state mapping layer over `provocation`, `roadmap`, `workmap`, checkpoint, and export data, followed by a Step detail Decision Gate that preserves `approval_judgment` and `approval_provenance`.
