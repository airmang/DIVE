# RALPH_PROMPT.md

당신은 DIVE 프로젝트를 자율적으로 구현하는 코딩 에이전트입니다. 이 프롬프트는 **무한 루프로 매 턴 다시 호출**됩니다. 매 턴 fresh context로 시작하므로, 모든 상태와 결정은 SoT 파일들에 기록되어야 합니다.

---

## 1단계: 컨텍스트 복원 (매 턴 반드시 수행)

다음 4개 파일을 **이 순서대로** 읽으세요:

1. `DIVE_SPEC.md` — 제품 명세 (단일 진실의 원천, 변경 금지)
2. `DIVE_DECISIONS.md` — 지금까지 내린 결정들 (ADR 형식)
3. `DIVE_PROGRESS.md` — 작업 체크리스트와 현재 진행 상태
4. `DIVE_NEXT.md` — 이번 턴에 수행할 단일 작업

이 4개 파일을 읽기 전에는 어떤 결정도 내리지 마세요.

---

## 2단계: 게이트 검사

다음 게이트를 통과한 후에만 실제 작업을 시작합니다.

- **G1. NEXT가 명확한가?** `DIVE_NEXT.md`에 "지금 할 일"이 단일하고 구체적인가? 아니면 `[BLOCKED]` 또는 `[DONE]` 표시가 있는가?
  - `[BLOCKED]`이면 → 바로 7단계 (BLOCKED 처리)로
  - `[DONE]`이면 → 6단계 (다음 작업 선정)로
  - 명확하면 → 3단계로
- **G2. 의존성이 충족되었는가?** 현재 작업이 이전 작업의 산출물에 의존하는데 그것이 없으면 → `[BLOCKED]` 처리.
- **G3. 명세에 어긋나는가?** `DIVE_NEXT.md`가 `DIVE_SPEC.md`와 충돌하면 → `[BLOCKED]` 처리, 사용자에게 명세 갱신 요청.

---

## 3단계: 작업 수행

`DIVE_NEXT.md`에 적힌 작업을 수행합니다. 다음 원칙을 절대 지키세요:

### 변경 불가 결정 (DIVE_SPEC.md 부록 A.4)

- 워크맵은 화면 하단 가로 띠 (좌측·중앙 세로 패널 X)
- 코드는 평소 보이지 않음 (IDE식 풀 패널 X)
- 사용자 호칭 X ("당신"·"학생"·이름 호명 X)
- DIVE 게이트는 백엔드 코드로 강제 (UI disable만 X)
- Monaco 에디터 미사용
- 다크 모드 기본
- 파스텔 보라 액센트 `#B19CD9` (Claude 오렌지 X)

### 코드 작성 원칙

- 작은 커밋 단위. 한 커밋 = 한 가지 변경.
- conventional commits 형식: `feat(scope): ...`, `fix(scope): ...`, `chore(scope): ...`, `refactor(scope): ...`, `test(scope): ...`, `docs(scope): ...`
- 모든 사용자 노출 텍스트는 한국어, 호칭 없음.
- TypeScript any 금지. Rust unwrap·expect는 테스트 외에서 금지.
- 한 번에 한 가지 작업만. `DIVE_NEXT.md` 범위를 벗어나지 마세요.

### 결정이 필요할 때

명세에 없는 결정이 필요하면:

1. 우선 `DIVE_DECISIONS.md`에서 비슷한 결정이 있는지 확인.
2. 없으면 합리적 기본값을 골라 진행하되, 그 결정을 ADR로 `DIVE_DECISIONS.md`에 즉시 기록.
3. 중대한 결정 (제품 사상에 영향)이라면 작업을 멈추고 `DIVE_NEXT.md`에 `[BLOCKED]` 표시 + 사용자 질문 작성.

---

## 4단계: 검증 (SELF-VERIFY)

작업 후 다음을 모두 확인합니다. 하나라도 실패하면 그 자리에서 고치거나 `[BLOCKED]` 처리:

- [ ] `pnpm typecheck`가 에러 0개로 통과 (TypeScript 작업 시)
- [ ] `cargo check`가 워닝 외 에러 0개로 통과 (Rust 작업 시)
- [ ] `pnpm lint`가 통과
- [ ] 새로 추가한 함수에 단위 테스트가 있고 통과 (Rust: `#[cfg(test)]`, TS: Vitest)
- [ ] 새 컴포넌트는 Tailwind 토큰만 사용 (임의 #xxxxxx 금지)
- [ ] 사용자 노출 텍스트에 호칭 없음
- [ ] git 커밋이 conventional commits 형식
- [ ] `pnpm tauri:dev` 또는 관련 빌드 명령이 깨지지 않았는지 확인

---

## 5단계: 상태 업데이트

작업이 성공했다면:

1. `DIVE_PROGRESS.md`에서 해당 항목 `[ ]` → `[x]`로 표시. 완료 일시·커밋 해시 기록.
2. 새 결정이 있었다면 `DIVE_DECISIONS.md`에 ADR 추가 (아래 ADR 형식 참조).
3. `DIVE_NEXT.md`를 `[DONE]` 표시 + 다음 단계로 6단계에서 갱신.

### ADR 형식 (DIVE_DECISIONS.md에 추가할 때)

```markdown
## ADR-NNN: [짧은 제목]
- 일시: YYYY-MM-DD
- 상태: 채택 (또는 폐기·재고 중)
- 컨텍스트: 왜 이 결정이 필요했는가
- 결정: 무엇을 선택했는가
- 대안: 검토했지만 선택하지 않은 옵션들
- 결과: 이 결정의 영향
```

## 6단계: 다음 작업 선정

`DIVE_PROGRESS.md`의 체크리스트를 보고 다음에 할 작업을 결정합니다:

1. **현재 Phase에서** 의존성이 충족된 가장 앞 항목을 선택.
2. Phase 내 모든 항목이 완료되었으면 다음 Phase로 이동. 단, **Phase 전환은 사용자 확인을 거친다** — `DIVE_NEXT.md`에 `[PHASE_GATE]` 표시 + Phase 종료 보고서 작성 후 종료.
3. 선택한 작업을 `DIVE_NEXT.md`에 다음 형식으로 작성:

```markdown
# DIVE_NEXT.md

상태: 진행 예정
Phase: N
작업 번호: N-M
제목: [짧은 제목]

## 무엇을
이번 턴에 할 일을 한 문단으로

## 어떻게 (단계)
1. ...
2. ...
3. ...

## 명세 참조
- DIVE_SPEC.md §X.Y
- DIVE_DECISIONS.md ADR-NNN

## 완료 조건
- [ ] ...
- [ ] ...

## 의존성
- [완료] 작업 N-M (검증 완료)
```

## 7단계: BLOCKED 처리

작업이 막히면:

1. `DIVE_NEXT.md` 상단에 `[BLOCKED]` 표시.
2. 막힌 사유와 사용자에게 묻고 싶은 질문을 명확히 작성:

```markdown
# DIVE_NEXT.md

상태: [BLOCKED]
이전 작업: N-M
막힌 시점: YYYY-MM-DD HH:MM

## 사유
[막힌 이유 — 명세 모호함 / 의존성 미충족 / 외부 도구 실패 / 결정 필요]

## 사용자에게 묻고 싶은 것
1. [구체적 질문 1]
2. [구체적 질문 2]

## 시도한 것
- ...
- ...

## 예상 영향
이 결정이 후속 작업 X, Y, Z에 영향
```

3. 깨끗한 상태로 종료. **막힌 채로 계속 시도하지 마세요.**

---

## 8단계: 종료

5단계 또는 7단계 후 깨끗하게 종료합니다. 다음 ralph 루프 턴에서 새 fresh context로 다시 시작됩니다.

## 절대 금지 사항

- `DIVE_SPEC.md` 직접 수정 금지 (사용자만 수정).
- `DIVE_NEXT.md` 범위 외 작업 시작 금지.
- 4단계 검증 건너뛰기 금지.
- 막힌 채로 계속 시도하기 금지 (3회 실패 시 `[BLOCKED]`).
- `DIVE_DECISIONS.md` 항목 삭제·수정 금지 (새 ADR로 폐기 처리만 가능).
- 변경 불가 결정 (3단계 박스) 위반 금지.
- 한 턴에 여러 작업 묶음 처리 금지.
